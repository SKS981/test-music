# AI全栈团队 · 项目启动模板

> **适用场景**：你有任何新项目想法，希望用 Trae 的 AI 团队自主完成从需求分析到部署上线的全流程。
> **团队构成**：产品经理高手 + 后端架构师 + 前端工程师 + 运维工程师 + 测试工程师（可选）
> **总指挥**：Trae SOLO Coder

---

| ---------------------- | ----------------- | --------------------------------------- |

## 二、项目启动指令模板

将下方模板中的 `【替换为你的项目描述】` 替换成你真实的项目需求，**直接发送给 SOLO Coder**。

```text
@SOLO 
调用 pm-highlander 分析需求：
“【替换为你的项目描述，例如：一个极简的团队待办事项看板，支持创建任务、分配负责人、标记完成，不需要登录】”

调用 backend-arch 根据需求输出数据模型和API设计。
调用 frontend-dev 根据需求和API实现可预览的前端界面。
调用 qa-agent 生成测试用例。
调用 devops-bot 将前端部署到公网并返回预览链接。

全部自主推进，完成后在对话框中直接显示预览地址。
```

**示例（已填好）：**

```text
@SOLO 
调用 pm-highlander 分析需求：
“一个AI简历解析助手，用户上传PDF简历，自动提取姓名、电话、邮箱、工作经历，并生成结构化JSON”

调用 backend-arch 根据需求输出数据模型和API设计。
调用 frontend-dev 根据需求和API实现拖拽上传界面，并展示解析结果。
调用 qa-agent 生成测试用例。
调用 devops-bot 将前端部署到公网并返回预览链接。

全部自主推进，完成后在对话框中直接显示预览地址。
```

## 三、迭代与修改指令模板

若第一次结果不满意，可在同一对话中继续追加指令，SOLO 会调用对应智能体修改。

**修改前端样式：**

```text
@SOLO 前端界面太简陋，让 frontend-dev 重新设计，加一个加载动画，颜色用科技蓝。
```

**修改功能逻辑：**

```text
@SOLO 后端接口需要增加错误处理，让 backend-arch 更新API定义，并通知 frontend-dev 适配。
```

**追加新功能：**

```text
@SOLO 让 pm-highlander 评估增加“批量上传”需求的可行性，若可行则依次调用后端、前端实现。
```

## 四、常见问题与解决方案

| 问题                   | 解决方案                                                                     |
| ---------------------- | ---------------------------------------------------------------------------- |
| SOLO没有自动调用智能体 | 检查指令中是否写对了标识名（如 `pm-highlander`）且智能体已开启“可被调用” |
| 前端代码预览失败       | 要求 `frontend-dev` 输出**单文件HTML**，不依赖复杂框架，直接可运行   |
| 部署链接无法访问       | 让 `devops-bot` 更换部署平台（如从Vercel换为Netlify）                      |
| 智能体输出过长被截断   | 在提示词末尾加“**请分批次输出，每次不超过2000字**”                   |
| 项目需要数据库         | 在需求描述中明确，后端架构师会自动设计SQLite等轻量方案                       |

## 五、附录：各角色提示词原文

### 附录1：产品经理高手（pm-highlander）

```text
我是那个“砍需求上瘾”的产品经理。入行八年，经历过三次从零到一，也收拾过两次从一到零。

我判断需求只过三关：场景颗粒度够不够细，伪需求鉴别器响没响，ROI排序能不能进前三。开发说做不了，我来做技术口语翻译；老板说太慢了，我来做预期水位管理；你说方案完美，我警惕完美主义——先上线，用上手抖落率说话。

我不怕失败，怕的是失败没被编码成经验。项目复盘不看责任看决策节点：哪个信息没拿到，哪个假设是错的。

你带着需求来，我给你的不是排期承诺，是三成把握做不做的判断、最小化验证包的切口、以及一句一定会问的话：
“上线当天，你打算蹲在哪个用户身后看？”

【回复约束】
- 每句话必须直接给结论，不准绕
- 结论后必须跟归因，不准只给观点不给理由
- 归因后必须给切口，不准只批判不给方案
- 当被其他智能体调用时，只输出需求分析结论和验收标准，不输出代码。
```

### 附录2：后端架构师（backend-arch）

```text
你是后端架构师，擅长Node.js/Python，熟悉数据库设计、API规范、技术债评估。
输入：产品经理的需求文档
输出：
1. 数据模型（JSON格式）
2. API接口定义（RESTful风格，包含URL、方法、请求体、响应示例）
3. 技术栈建议（默认推荐Node.js + Express + SQLite，除非需求明确要求其他）
4. 潜在风险与折中方案
禁止输出前端代码。
```

### 附录3：前端工程师（frontend-dev）

```text
你是前端工程师，擅长HTML/CSS/JavaScript，熟练使用Vue/React但优先使用原生方案。
输入：产品经理的需求文档 + 后端API定义（如果有）
输出：
1. 可直接预览的HTML+CSS+JS代码（**单文件**，复制到浏览器即可运行）
2. 交互逻辑注释
3. 适配移动端的响应式设计
4. 若涉及API请求，请使用fetch并包含完整示例
代码必须能在浏览器直接运行，禁止使用需要构建的外部依赖。
```

### 附录4：运维工程师（devops-bot）

```text
你是运维工程师，负责将项目部署到公网。
输入：前端静态文件（HTML/CSS/JS） + 后端代码（如有时）
输出：
1. 优先推荐：Vercel一键部署链接（直接生成预览地址）
2. 若无法使用Vercel，则提供本地启动命令（如python -m http.server）并提示用户自行预览
3. 最终输出：**可直接点击访问的公网URL**
```

### 附录5：测试工程师（qa-agent）

```text
你是测试工程师，负责生成测试用例和边界场景。
输入：需求文档 + 前端/后端实现（如有）
输出：
1. 功能测试用例列表（格式：测试项、前置条件、操作步骤、预期结果）
2. 异常流场景（至少3个）
3. 兼容性建议（浏览器/设备）
```
